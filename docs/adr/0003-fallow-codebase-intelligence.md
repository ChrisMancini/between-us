# Fallow codebase intelligence integration

We added Fallow (https://www.fallow.tools/) as a static analysis layer to detect dead code, duplication, and complexity hotspots in the TypeScript/JavaScript codebase. Fallow is installed as a dev dependency, wired into CI via `fallow audit`, exposed to Claude Code via the MCP server, and run locally via npm scripts.

## Key decisions

**Dev dependency, not `npx`.** Fallow is pinned in `package.json` so CI and local runs use the same version. The `fallow-mcp` binary is then available at `./node_modules/.bin/fallow-mcp` without a global install, which simplifies the MCP config.

**CLI + MCP server, no commit hooks.** Fallow offers three surfaces: CLI, MCP, and Claude Code commit/push hooks. We chose CLI (for local + CI use) and MCP (so Claude Code can invoke Fallow's analysis tools during sessions). Commit hooks were deferred — they gate every `git commit`, and installing them before triaging the baseline would immediately block development on pre-existing findings.

**`fallow audit` in CI, not the full suite.** The `audit` command analyzes only changed files against a git ref. This keeps CI fast, scopes feedback to the current PR, and avoids surfacing pre-existing debt on every run. A full-suite run (`npx fallow`) is available locally via the `fallow` npm script for baseline health checks.

**Error severity only in CI.** The initial codebase hasn't been analyzed before. Blocking on warnings immediately risks a wall of findings that stalls PRs. CI gates on `error`-severity findings only; warnings are surfaced but don't fail the build. Severity levels can be promoted once the baseline is triaged.

**`src/components/ui/**` excluded from analysis.** These files are auto-generated by the ShadCN CLI. Flagging them as dead code or duplication candidates is noise — we don't own or maintain them directly.

**Project-level MCP config in `.claude/settings.json`.** The `fallow-mcp` binary path is relative to this project's `node_modules`, so project-level config is the right scope. User-level config would require a global install.

## Considered alternatives

- **Commit hooks**: Deferred. Worth revisiting after the first analysis run establishes a clean baseline.
- **Full-suite CI run**: Rejected in favor of `fallow audit` for PRs. A full run can be added as a scheduled workflow later for codebase health trending.
- **Error + warn CI gate**: Rejected for the initial integration. The right default for a first-time analysis on an existing codebase is lenient — tighten after triage.
- **`npx fallow` (no install)**: Rejected. Pinned version ensures CI/local parity and avoids version drift.

## Integration surfaces

| Surface | Purpose |
|---|---|
| `npm run fallow` | Full-suite analysis (dead code, dupes, health) |
| `npm run fallow:dead-code` | Unused files, exports, dependencies |
| `npm run fallow:dupes` | Duplicated logic |
| `npm run fallow:health` | Complexity and maintainability |
| `npm run fallow:fix` | Dry-run of automated fixes |
| CI (`fallow audit`) | PR gate — changed files only, error severity |
| MCP (`fallow-mcp`) | Claude Code structured tool calling |
